# AIBidder 模块体系重构 — 差距分析与修订计划

> 基于用户提出的 6 模块需求，与当前项目需求文档 + 代码实现进行全面对照。

## 一、用户目标模块 vs 现有系统对照

用户提出的核心诉求是：**以无 IT 背景的投标工程师为目标用户，重塑产品操作流程。** 下表逐模块对照当前需求文档与代码实现的覆盖情况。

### 模块映射总览

| # | 用户模块 | 前端名称 | 需求文档对应 | 代码实现状态 | 差距等级 |
|---|---|---|---|---|---|
| 1 | **投标资料库** | 投标资料库 | §3.1 文件上传与处理 + §6 知识模型 | ⚠️ 部分实现 | 🟡 中 |
| 2 | **标书分析** | 标书分析 | §3.2 招标文件拆解 | ❌ 未实现（仅有数据表） | 🔴 高 |

## 二、逐模块差距详解

### 模块 1：投标资料库

> 用户需求：将历史投标文件、优秀标书、公司资质及业绩和设施设备、人员资质及业绩等文件纳入投标资料库中，可查询入库资料基本情况，以及审查敏感信息清洗。

| 子功能 | 需求文档覆盖 | 代码实现 | 差距 |
|---|---|---|---|
| 文件上传（PDF/DOCX） | ✅ §3.1 | ⚠️ 上传通道存在但 PDF 解析为假实现 | PDF 解析不可用 |
| 文件分类存储 | ✅ §6.1 定义了 9 种库分类 | ⚠️ `KnowledgeBaseEntry` 有 `library_type` 字段 | 分类体系骨架已有 |
| 资料查询列表 | ✅ 隐含需求 | ⚠️ `list_library_entries` API 已有 | 缺少筛选/搜索能力 |
| 敏感信息清洗 | ✅ §2.6 泄漏校验 | ⚠️ `historical_leakage_checker` + `risk_marker` 存在 | 仅针对历史标书，**未覆盖通用资料库** |
| 企业事实表（资质/人员/设备/业绩） | ✅ §6.1 | ⚠️ 模型已建（`Qualification`, `PersonnelAsset`, `EquipmentAsset`, `ProjectCredential`） | 模型存在但**无对应 API 和 UI** |

**关键差距**：
1. PDF 解析器是假实现，上传 PDF 招标文件无法正确提取内容
2. 企业事实表有数据模型但无增删改查 API
3. 前端无独立资料库管理界面（当前嵌在单文件 `page.tsx` 内的 workbench tab）
4. 敏感信息清洗仅针对历史标书复用单元，未扩展到全库

---

### 模块 2：标书分析（招标文件拆解）

> 用户需求：将 PDF 招标文件上传后分解为基本信息、资格要求、评审标准、投标要求、废标条款、提交清单、合同风险 7 类内容，并可与原文对照显示。

| 子功能 | 需求文档覆盖 | 代码实现 | 差距 |
|---|---|---|---|
| 招标文件上传 | ✅ §3.2 | ⚠️ 复用文档上传通道 | PDF 解析不可用 |
| 7 类结构化拆解 | ✅ §3.2 拆解产出 | ❌ 无拆解引擎 | **核心缺失** |
| 拆解结果持久化 | ✅ 计划中 | ⚠️ `TenderRequirement` + `RequirementConstraint` 表已建 | 仅有数据表，无写入逻辑 |
| 原文对照显示 | 用户**新增需求** | ❌ 需求文档未提及 | **新增需求** |
| `DecompositionRun` 管理 | ✅ 计划中 | ⚠️ 表和 CRUD API 已有 | 仅占位，无真正拆解流程 |

**关键差距**：
1. 无 AI 拆解引擎（需调用 LLM 对招标文件进行结构化提取）
2. 用户新增了**原文对照显示**需求（需实现 PDF 渲染 + 高亮锚点）
3. 7 类拆解输出与需求文档定义的输出有差异（需求文档更偏工程分类，用户更偏投标工程师习惯）

> [!IMPORTANT]
> 用户定义的 7 类输出（基本信息、资格要求、评审标准、投标要求、废标条款、提交清单、合同风险）与需求文档中的分类（资格要求、技术要求、商务要求、评分要点、废标条款、必答项、硬约束快照）有偏差。应以用户定义为准进行调整。

### 模块 3：标书生成

> 用户需求：以招标文件拆解和资料库为约束，AI 先生成框架，框架批准后分章节编写标书初稿。

| 子功能 | 需求文档覆盖 | 代码实现 | 差距 |
|---|---|---|---|
| 框架生成 | ✅ §8.2 plan 阶段 | ❌ 无实现 | **核心缺失** |
| 框架人工审批 | 用户**新增需求** | ❌ | **新增需求** |
| 分章节编写 | ✅ §12.3 | ❌ 无实现 | **核心缺失** |
| 证据绑定 | ✅ §12.3 | ⚠️ `SectionEvidenceBinding` 表已建 | 仅有数据模型 |
| 生成任务管理 | ✅ 计划中 | ⚠️ `GenerationJob` + CRUD API 已有 | 仅占位 |

**关键差距**：
1. 完全没有 AI 编排器（plan → retrieve → draft → verify）
2. 用户新增了**框架审批**环节（先看大纲，再写正文）
3. 需要异步 Worker 支持长时间生成任务（当前 Worker 为空壳）

---

### 模块 4：标书评审

> 用户需求：对标书初稿进行分析、评审、模拟评分、废标分析，不合格打回重写，最终形成合格标书。

| 子功能 | 需求文档覆盖 | 代码实现 | 差距 |
|---|---|---|---|
| 多维评审 | ⚠️ 需求文档§8.2 verify 阶段提到但未独立为模块 | ❌ | **核心缺失** |
| 模拟评分 | ❌ 需求文档未提及 | ❌ | **新增需求** |
| 废标分析 | ✅ §3.2 starred + invalid-bid | ❌ | **核心缺失** |
| 打回重写流程 | 用户**新增需求** | ❌ | **新增需求** |
| `ReviewRun` + `ReviewIssue` | ✅ 计划中 | ⚠️ 表和 CRUD API 已有 | 仅占位 |

**关键差距**：
1. 需求文档将"验证"作为生成的子步骤，**用户将其升级为独立模块** — 这更符合投标实务
2. 新增**模拟评分**功能，需根据评审标准进行自动打分预测
3. 新增**打回重写**闭环（评审 → 生成的反馈回路）

> [!IMPORTANT]
> 标书评审独立成模块是用户的一个重要设计决策。在投标实务中，标书编写和评审是不同角色（编写人 vs 审查人）在不同阶段完成的工作，将其分开是正确的。

| 维度 | 当前状况 | 目标状态 | 差距 |

## 四、需求文档本身的修订建议

用户的 6 模块体系更贴近投标工程师的实际操作习惯，应调整需求文档的以下章节：

| 原始需求文档章节 | 修订方向 |
|---|---|
| §3.1 文件上传与处理 | 重新定位为「投标资料库」的后端能力 |
| §3.2 招标文件拆解 | 调整输出分类为用户定义的 7 类 |
| §3.3 标书生成 | 增加「框架审批」环节 |
| 无独立评审模块 | **新增** §3.X 标书评审模块（含模拟评分、废标分析、打回重写） |
| §3.4 格式化 → §3.5 排版定稿 | 名称和操作流程调整 |
| §3.5 项目管理 | 拆分：项目管理能力保留在底层，新增「标书管理」作为用户操作模块 |
| §16.4 前端规划提示 | 更新页面列表为 6 模块体系 |

## 七、总结

### 核心修订要点

1. **模块体系**：从技术导向的 5 模块（上传/拆解/生成/格式化/项目管理）调整为用户导向的 6 模块（资料库/分析/生成/评审/排版/管理）
2. **新增功能**：模拟评分、原文对照、框架审批、打回重写、优秀标书回灌
3. **前端重构**：从单文件 console 重构为 6 模块独立页面
4. **命名调整**：「招标解析」→「标书分析」、「标书检测」→「标书评审」
5. **基础设施**：PDF 解析、PostgreSQL FTS、Celery Worker 必须在第一阶段修复

### 预估总工作量

| 阶段 | 预估时间 | 前置依赖 |
|---|---|---|
| Phase R1 基础补强 + 资料库 | 2-3 周 | 无 |
| Phase R2 标书分析 | 2-3 周 | R1 |
| Phase R3 标书生成 | 3-4 周 | R2 |
| Phase R4 标书评审 | 2-3 周 | R3 |
| Phase R5 排版定稿 + 标书管理 | 2-3 周 | R4 |
| **总计** | **11-16 周** | |

> [!WARNING]
> 当前项目最大的技术风险仍然是 **PDF 解析质量**。建议在 Phase R1 优先验证解析方案在跨页表格、评分表等投标典型场景上的表现，否则后续所有模块（特别是标书分析）的质量将受到根本性制约。
