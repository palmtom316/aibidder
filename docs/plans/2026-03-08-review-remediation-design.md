# 评审整改设计说明

**目标**

将 `docs/2026-03-07-aibidder-review-report.md` 中仍然有效的整改项收敛为一组可交付的工程变更，使当前仓库从“多数关键短板已识别”推进到“关键链路可运行、可验证、可追踪”。

**范围**

本轮整改覆盖以下仍未完全闭环的事项：

1. 存储后端抽象化，并补齐 MinIO 兼容路径。
2. PDF 解析链路升级为“文本提取 + 可选 OCR 适配器 + 结构化错误码”。
3. 历史标书检索补齐文本搜索能力，并复用 PostgreSQL FTS。
4. `verification_issues` 从仅建表升级为真实写入与查询闭环。
5. 章节证据绑定补齐外键级关联，而非仅保存锚点文本。
6. SSE 从单次快照升级为轮询式持续事件流。
7. 前端工作台将超大 `page.tsx` 按模块视图拆分，显著降低单文件复杂度。

**不在本轮范围**

1. 完整引入 Tailwind / shadcn 设计系统。
2. 将后端重构为完整 `app/domains/*` 目录体系。
3. 接入真实外部 OCR 服务并依赖线上密钥进行端到端验收。
4. 完整微服务化拆分。

## 方案备选

### 方案 A：一次性大重构

直接同时重做存储、解析、搜索、前端架构和样式体系。

- 优点：一步到位。
- 缺点：变更面过大，回归风险高，不适合当前主分支连续整改。

### 方案 B：分层补齐关键闭环（推荐）

先补运行时关键能力与数据闭环，再做前端拆分，保持接口兼容。

- 优点：风险可控；现有测试可逐步增强；适合当前代码基础。
- 缺点：仍会保留部分架构债务，如未完全 domain 化。

### 方案 C：只做审计文档同步

更新文档，不做代码整改。

- 优点：成本最低。
- 缺点：不能解决真实问题，不满足用户要求。

**推荐方案：B**

## 目标设计

### 1. 存储层

新增 `StorageBackend` 协议与统一入口：

- `LocalStorageBackend`：保留当前本地文件系统行为。
- `MinioStorageBackend`：通过可选依赖的 S3 兼容客户端工作；未安装客户端时给出清晰错误。
- 上层调用方不再直接操作 `Path(settings.storage_root)`，而统一通过存储接口保存、读取、下载。

### 2. PDF 解析层

解析顺序调整为：

1. `PyMuPDF` 提取文本与页面。
2. `pypdf` 作为退化文本提取。
3. 当页面缺少可用文本且启用 OCR 时，走 OCR 适配器。
4. 最终失败时返回结构化错误码写入 `parse_log`。

OCR 适配器本轮采用“接口先行、客户端可选”的方式：

- 默认关闭，不影响本地开发。
- 通过配置启用。
- 单测使用 monkeypatch 验证行为，不依赖真实网络。

### 3. 搜索层

- 证据搜索继续使用 PostgreSQL FTS，但优先使用已生成的 `fts_vector` 列。
- 历史标书复用搜索新增 `q` 参数，支持 PostgreSQL FTS 与 SQLite `ilike` fallback。
- API 保持向后兼容：原过滤条件仍可用。

### 4. 校验 / 评审闭环

- 在生成后或评审时写入 `verification_issues`。
- `review_issues` 继续承担面向用户的问题单角色。
- `verification_issues` 用于保存系统级校验发现，如内容过短、缺少证据绑定、合同风险提醒等。
- 生成流程落库 `SectionEvidenceBinding` 时尽量绑定到 `evidence_unit_id` / `historical_reuse_unit_id`，保留锚点文本作为补充字段。

### 5. SSE

- 将单次返回改为轮询式 `text/event-stream`。
- 每次请求在短连接内持续输出当前状态，直到任务终态或超时。
- 不引入额外 broker / pubsub，避免扩大基础设施范围。

### 6. 前端拆分

- 保持现有路由不变：`/workspace/[module]` 继续生效。
- 将 `src/frontend/app/page.tsx` 拆为：
  - 共享状态/数据 hook
  - 模块级视图组件（资料库、标书分析、标书生成、标书评审、排版定稿、标书管理）
  - 壳层页面只负责路由与布局编排
- 本轮不引入新的状态库，避免扩大变更面；先通过模块组件拆分降低耦合。

## 错误处理

- 文档解析失败统一写入 `parse_log.code` / `parse_log.message`。
- MinIO 客户端缺失、OCR 未配置、对象不存在等错误提供稳定错误码。
- SSE 轮询遇到对象不存在时返回标准 `404`。

## 测试策略

- 后端采用 TDD：为存储、历史搜索、PDF OCR fallback、verification issues、evidence bindings、SSE 新增或扩展测试。
- 前端暂无单测体系，本轮以 `npm run build` 作为回归验证，并尽量保持组件接口稳定。
- 最终执行后端测试 + 前端构建作为收尾验证。
